🔴 Backend — Errores críticos

1 · Los genes core no se investigan correctamente

Bug get_pubmed_details — f-string rompe la query
La línea f'("{2026-years}"[Date...])' evalúa la expresión aritmética 2026 - 10 = 2016 en tiempo de parseo, pero si years fuera 0 daría un número incorrecto. Además, si el resultado es un número y no un string, PubMed rechaza la query silenciosamente devolviendo 0 resultados.

Solución: calcular el año antes: start_year = 2026 - years y usar f'("{start_year}"[Date...])'.
Bug Solo se investigan los primeros 40 genes, pero enriquecimiento usa hasta 500
gene_details se llena con common_all[:40]. El frontend intenta data.gene_details[gene] para todos los common_genes, lo que causa TypeError silencioso en JS (undefined) para genes del 41 al N.

Solución: subir el límite a al menos 100, o usar el mismo slice que common_genes.
Bug get_targetscan_genes_local — nunca encuentra el archivo
Busca el archivo como RAW_DATA_DIR / f"{mirna}.txt", donde mirna ya está normalizado a hsa-miR-33a-5p. Pero los archivos en el repo se llaman igual (hsa-miR-33a-5p.txt). El problema es que RAW_DATA_DIR apunta a ../../../repo_tesis/data/raw, que es una ruta relativa al ejecutable, no garantizada en producción. Si el backend corre desde otra carpeta, devuelve set() siempre y los genes core son vacíos.

Solución: usar Path(__file__).parent / "../../.." / "repo_tesis/data/raw" con .resolve() para obtener la ruta absoluta.
Bug await get_kegg_scientific_summary dentro de async with — se llama dos veces
En run_pipeline, get_kegg_scientific_summary está decorada como async pero no hace ningún I/O — devuelve un string local. Se puede hacerla síncrona, eliminando un await innecesario que puede causar warnings en algunas versiones de Python.

2 · Problemas de lógica y datos

Lógica consensus_mode = "n-2" puede devolver genes de un solo miRNA
Si hay 3 miRNAs y threshold = max(1, len-2) = 1, el sistema devuelve todos los genes que aparezcan en cualquier base de datos, no los consensuados. Esto infla artificialmente los resultados sin validez científica real.
Datos score del gen se calcula como porcentaje pero se trata como puntos
score = (suma/total)*40 + 30 + 30 — el resultado va de 0 a 100, pero en el frontend se muestra como "Índice de confianza del 70.5%". Si el score ya es sobre 100, el texto debería decir puntos, no porcentaje, o normalizarse bien.
API La clave PubMed API está hardcodeada en el código fuente
PUBMED_API_KEY = "f6cba57..." — nunca debe ir en el código. Moverla a una variable de entorno: os.getenv("PUBMED_API_KEY").

🔵 Frontend — Problemas de diseño y editor

Conflicto CSS Dos hojas de estilo se contradicen
style.css define el tema oscuro (--bg-dark: #0d1117, sidebar de 280px). El <style> dentro del <head> redefine sidebar a 240px y card headers con color claro. Hay conflictos de especificidad que hacen que algunos elementos se vean distintos dependiendo del orden de carga. Unificar todo en un solo archivo CSS.
Editor La toolbar del editor no funciona sobre contenido generado dinámicamente
execCmd('bold') usa document.execCommand, que está deprecado y no funciona en Firefox moderno. Además, los elementos contenteditable se crean dentro del canvas del informe después de que se ejecuta el análisis — la toolbar sólo afecta la selección activa, y si el usuario no ha seleccionado texto primero, no hace nada visible.
UX Los "gene pills" clickeables no tienen estado de carga visible
Al hacer clic en un gen, el sidebar aparece con texto genérico y luego cambia. No hay indicador de qué gen se está cargando, ni skeleton/loader diferenciado. En redes lentas, el usuario puede clicar varias veces pensando que no funcionó.
Diseño El dashboard mezcla dos identidades visuales
El sidebar usa el tema oscuro tipo GitHub Dark (#0d1117), mientras que las cards usan fondo blanco con bordes grises claros. El resultado es una app que parece ensamblada con piezas de dos proyectos distintos. El viz-container fuerza background: #fff dentro del tema oscuro, creando un contraste brusco.
Diseño El editor de informe no tiene feedback cuando está vacío
Si el usuario navega al editor antes de ejecutar el análisis, ve páginas en blanco sin ninguna instrucción. Se necesita un estado vacío claro: "Ejecuta un análisis primero para ver el informe generado."
Tipografía La fuente JetBrains Mono no está cargada
El CSS referencia font-family: 'JetBrains Mono', monospace en .logo y .status-badge, pero en el <head> solo se cargan Spectral, Inter e IBM Plex Mono desde Google Fonts. El logo y el badge caen al monospace del sistema.

Prioridad de corrección sugerida

1. Crítico Ruta absoluta de RAW_DATA_DIR → los genes core siempre devuelven vacío

2. Crítico F-string en get_pubmed_details → PubMed no retorna evidencias

3. Crítico Límite [:40] en gene_details → crash silencioso en frontend

4. Importante Unificar CSS (style.css vs <style> inline)

5. Importante Reemplazar execCmd por Selection API moderna

6. Mejora Mover API key a variable de entorno

7. Mejora Cargar JetBrains Mono o reemplazar por IBM Plex Mono (ya cargada)
